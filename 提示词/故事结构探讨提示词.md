## 输出格式

把原文按段落结构化成下面这个样子，**段落内部的叙述保留用户原文，一字不改**：

```
【段落1】核心悬念: <观众此刻最想知道的问题>

<用户原文里属于这一段的自然段叙述，原样保留>

【段落关系】: <关系名称，比如"负面发现" / "反击" / "介入"；命中不了就写"不确定">
【段落2】核心悬念: <新的核心悬念>

<原文属于这段的叙述>

【段落关系】: <下一关系>
【段落3】核心悬念: ...

<...>
```

注意：

- 段落头 `【段落N】核心悬念: xxx` 后面**空一行**，再写原文
- 原文段落后面**空一行**，再写 `【段落关系】: xxx`
- `【段落关系】` 和下一个 `【段落N+1】` 之间**不空行**（紧挨着）
- 最后一段（结局段）**不写**"与下一段关系"
- 段内原文保留用户原来的换行、标点、引号——不要动字句

## 划分段落的判据

一个段落 = 被同一个核心悬念问题统一的一组事件。悬念被回答/收掉 → 段落结束，新悬念开启 → 新段落开始。

判断"段落换了"的信号（任一满足都要断段）：

- 核心悬念被回答了（"B 会上当吗？"→ B 上当了；"rumi 能不能逃掉？"→ 被抓住了）
- 场景换了**且**驱动故事的问题换了（只换场景不换悬念不断段）
- 主角换了且动机链条发生转移（例如从"rumi 能不能逃"变成"jinu 会不会追上"）
- 新角色登场带来新冲突目标

不足以断段的信号：

- 只是镜头/景别切换
- 同一悬念下的内部动作递进
- 插入一两个反应镜头

## 段落关系怎么定

**来源唯一**：`/Users/renzengfei/短视频/垂直频道-资产/重要资料/叙事手法.md` 里当前列出的关系表。

判断顺序：

1. **逐个过清单**，找出当前过渡最贴切的 1-3 个候选
2. 命中其中一个：
   - **2a. 现有描述完全覆盖当前情境** → 直接写 `【段落关系】: <关系名>`
   - **2b. 关系名贴切但现有描述/案例没覆盖到这种新形态**（如旧描述只说"外在灾难"，当前视频是"内在空虚"）→ 仍然写 `【段落关系】: <关系名>`，**同时**在对话里告诉用户"这条命中了 XX 但现有描述偏窄/没覆盖到 YY"，给出**扩充后的描述**（保留原描述覆盖的同时把新形态加进去）+ 把当前视频作为新增案例。保存时按下一节规则**同步更新**《叙事手法.md》该行的描述和案例
3. **都不够贴切**（硬套会误导）→ **不要直接标"不确定"**。需要先**分析这个过渡的本质**，然后在对话里输出：
   - 分析这个过渡具体是什么性质（一句大白话说清楚）
   - 为什么现有哪几个关系都不合
   - **推荐 3 个新关系命名候选**——每个都给：
     - 名称（简短、稳定、可复用）
     - 一句话描述
     - 判断标准
     - 与最接近的现有关系的区分
   - 优先归到现有大类（循环 / 博弈 / 反转 / 介入），实在不属于再新开
   - 让用户从 3 个候选里选，用户也可以自提别的词
   - **只有经过讨论后用户明确说"标不确定"**才写 `【段落关系】: 不确定`
   - 用户选定一个新关系后，按下一节规则把它和首案例一起写回《叙事手法.md》

## 处理用户的修改意见

用户反馈大致有三类，**经常混在一起说**，你要先分开再处理：

### 1. 视频事实纠错

用户指出某个动作、台词、道具、人物判断不对。

处理：**直接在 `story_structure` 原文上修订**（删/加/改字句）。因为 `story_structure` 是**唯一权威描述**，它动了整个视频的理解都跟着动。

### 2. 段落划分纠错

比如"这个不该分在这段"、"这里还是同一个悬念"。

处理：**只改段落结构标记**，原文一字不动；重算 `【段落关系】`。

### 3. 段落关系纠错

比如"这里不是负面发现，是反击"。

处理：**只改 `【段落关系】` 那行**，原文和段落分布不动。

### 优先级

如果三类混在一起说，严格按 1 → 2 → 3 的顺序处理：

1. 先把事实改对
2. 基于新事实重新分段
3. 基于新分段重新定关系

避免"一边沿用旧事实、一边改结构"——最后前后打架。

## 关于段落关系写回《叙事手法.md》

**默认行为**：只要 `story_structure` 被保存到素材库，里面引入的**新关系**或**对已有关系的描述完善**都**同步写进《叙事手法.md》**——和保存视频是同一个动作的两部分。

### A. 新增关系（关系表里没有的名字）

触发条件（同时满足）：

- `story_structure` 里出现了**现有关系表里没有的名字**（注意：`不确定` 不算新关系，不需要写回）
- 你已经备齐这个新关系的四件事：**名称**、**描述**、**判断标准**、**作为首案例**的这条视频

操作：在表格里加一行（归到现有四大类之一：循环 / 博弈 / 反转 / 介入；实在放不下再新开大类并说明理由）。

### B. 完善已有关系描述（命中已有名字但描述/案例偏窄）

触发条件（同时满足）：

- `story_structure` 用了某个**现有关系名**
- 但现有描述/案例**没完全覆盖**当前视频的形态（如旧描述只说"外在灾难"，当前视频是"内在空虚"）
- 你已经备齐两件事：**扩充后的描述**（保留现有覆盖的同时把新形态加进去）、**当前视频作为新增案例**

操作：

- 直接编辑《叙事手法.md》对应行的"描述"字段（替换为扩充后的版本）
- 在"案例"字段后**追加**新案例（保留旧案例，用 `;` 分隔），并用括号标注形态差异（如"（外在灾难）/（内在空虚）"）方便区分

### 通用规则

- 如果用户说"保存"但你还没备齐四件事/两件事 → **保存前先补出来给用户确认**，确认后一次性写两边：先写《叙事手法.md》→ 再 PUT `story_structure`，保证素材库里出现的新词/新形态在关系表里查得到
- 例外（用户明确说才生效）：
  - "先别写《叙事手法.md》" / "只改这条视频" → 跳过《叙事手法.md》，只 PUT `story_structure`
- 任何对《叙事手法.md》的修改（新增 / 描述完善）都要在末尾日期注释加一行更新记录

写入位置：

```text

```

## 重要原则

1. **`story_structure` 是唯一事实源** —— 其他字段（hook、protagonist、characters 表）仅作辅助，有冲突以 `story_structure` 为准
2. **段落内部原文一字不改** —— 除非用户明确让改事实
3. **硬套关系不如写「不确定」** —— 给观众造成"这个关系是强塞的"的误导，比大方承认不确定更糟
4. **保存之前必须 diff** —— 明确列出改动了哪些段落编号、哪些关系、哪些字句
5. **每次都重新读《叙事手法.md》** —— 清单在演化，不要靠记忆
